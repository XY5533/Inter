# IntelliRAG 项目设计文档

## 一、场景

一家工业自动化企业，内部沉淀了 8 本工业机器人教材、产品手册、NLP 复习资料、员工简历等非结构化文档。工程师在日常工作中遇到问题时——比如"ABB 机器人的伺服电机选型注意事项""搬运编程的路径规划怎么做"——只能翻 PDF、Ctrl+F 关键词，运气好 10 分钟找到，运气不好翻半小时还找不到。如果问题在知识库范围之外（比如问天气、问股票），传统检索既不会判断"我没这个信息"，也可能返回一堆碎片让用户误以为这就是答案。

**核心矛盾：知识在那里，但人找不到——找到了也不确定是不是完整的。**

## 二、这个项目到底是什么

一个面向企业文档的**知识库问答 Agent**。不是简单的"检索文档→丢给 LLM 总结"的 RAG 流水线，而是让 Agent 自己决定：要不要搜、搜什么、搜完够不够、不够怎么办、什么时候该说"我不知道"。

**本质**：给 LLM 装上"手"（检索工具）和"脑"（记忆），让它在企业知识库这个封闭环境里，像一个初级技术顾问那样——先理解问题、查资料、思考、再回答。

## 三、整体链路

```
用户：伺服电机选型注意事项？
  │
  ├── 输入层：纯文本 / 语音转文本 / 图片转描述 / 视频抽帧汇总
  │                 │
  │            ┌────┴────┐
  │            │ 统一为文本 │
  │            └────┬────┘
  │                 ▼
  ├── ReAct Agent ─────────────────────────────────────┐
  │   │                                                 │
  │   │  Step 1: THOUGHT "需要查知识库"                  │
  │   │           ACTION [SEARCH_KNOWLEDGE 伺服电机 选型]  │
  │   │           OBSERVATION ← FAISS 返回 Top-3 片段     │
  │   │                                                 │
  │   │  Step 2: THOUGHT "信息够了，顺便看用户有没有偏好"   │
  │   │           ACTION [SEARCH_MEMORY 伺服电机 选型]     │
  │   │           OBSERVATION ← "用户关注 ABB 品牌"        │
  │   │                                                 │
  │   │  Step 3: THOUGHT "可以给出完整回答"                │
  │   │           ACTION [FINAL_ANSWER ...]              │
  │   │                                                 │
  │   └── 安全兜底：超 5 步 / 重复搜索 / 异常 → 降级标准RAG │
  │                                                     │
  ├── 记忆层 ──────────────────────────────────────────┘
  │   短期：最近 5 轮对话原文 → 超过 10 轮触发 LLM 摘要压缩
  │   长期：FAISS 语义检索用户偏好/纠正/身份 → 注入 Prompt
  │
  └── 回答用户
```

## 四、关键设计决策

### 决策 1：为什么不用 LangGraph/CrewAI，而是手写 ReAct？

**场景约束**：这是个人实习项目，时间 1 周，目标是理解 Agent 原理而非调包。

LangGraph 封装了完整的图调度，把 ReAct 变成一个黑盒——面试官问"ReAct 怎么实现的"，你只能说"我调了 LangGraph"。手写 ReAct 的核心逻辑只有 200 行：

- System Prompt 定义四工具 + 输出格式
- 正则解析 THOUGHT / ACTION 边界
- 循环执行 + Observation 注入下一轮

**代价**：格式解析不完美需要兜底（不标准输出直接当 FINAL_ANSWER 处理）；**收益**：面试能讲清每一层的设计。

### 决策 2：为什么记忆分两层而不是只记录对话历史？

**场景驱动**：工程师反复问同类问题（比如连续 3 次问伺服电机），Agent 应该记住"这人关注伺服电机"，下次主动优先检索相关内容。

如果只有短期记忆（滑动窗口对话历史），退出重进就丢了。如果把全部对话存下来，上下文会膨胀，Token 爆炸。

**设计**：
- 短期 = 最近 k 轮原文，保证当前对话连贯性
- 长期 = LLM 自动从对话中提取"值得记的"→ 存在 FAISS 里 → 语义检索召回

"值得记的"定义为四类：**偏好**（关注 ABB 品牌）、**身份**（机械工程师）、**纠正**（用户说"不对，EtherCAT 不是脉冲控制"）、**高频**（同一问题出现 3 次以上）。

**为什么不存关系数据库**：长期记忆的检索场景是"根据当前问题语义找相关记忆"，天然适合向量检索——和知识库检索是同一套基础设施（FAISS），零额外成本。

### 决策 3：为什么多模态图片不直接做向量检索，而是先转文字？

**场景**：用户上传一张伺服电机的接线图，问"这个怎么接线"。

**直接做多模态向量检索**：需要多模态 Embedding 模型（如 CLIP），把图片和文本映射到同一向量空间。问题——要换整套检索管线（FAISS 换多模态向量库），成本高；而且企业知识库本身只有文本，CLIP 对技术图表的效果未经充分验证。

**先转文字再检索**：图片 → 调 qwen-vl-max → "这是一张伺服电机接线图，标注了 U/V/W 三相和编码器接口..." → 这段文字作为用户问题的补充上下文 → 进文本 RAG 检索。检索链路完全不动，只在前端加一个预处理步骤。

**适用边界**：这个方案适合"图片作为问题补充"的场景，不适合"以图搜图"。如果产品要求是用一张产品照片搜索所有包含该产品的文档，那需要多模态向量检索——但这个项目的用户场景是用图片描述问题、用文字搜索答案，转文本方案完全够用。

### 决策 4：为什么选 FAISS 而不是 ChromaDB？

**不是技术偏好的问题，是 Windows 环境逼出来的。**

ChromaDB 在 Windows 中文环境下，onnxruntime C 扩展在写入向量时 SIGSEGV 崩溃，升级/降级 onnxruntime 版本均无效。FAISS-cpu 是纯 CPU 的 C++ 库，没有 onnxruntime 依赖，稳定性好得多。

**代价**：FAISS 不支持增删单条数据（需完全重建索引），C++ 后端不支持 Unicode 路径（需迁到 ASCII TEMP 目录）。**判断**：这个项目的知识库是批量重建的（不是实时增量），FAISS 的不足在当前场景下不构成瓶颈——ChromaDB 的崩溃是致命问题。

## 五、边界与局限

| 局限 | 原因 | 什么场景下不够 |
|------|------|---------------|
| 单用户、无并发 | Streamlit 单进程 | 多部门同时使用时需换 FastAPI + 消息队列 |
| FAISS 不支持增量 | C++ 后端设计 | 知识库需要实时更新时需换 Milvus/Qdrant |
| ReAct Prompt 硬编码 | 手写实现 | 需要动态增减工具时需换 LangGraph |
| 无流式输出 | ChatTongyi 不支持 streaming | 体验不如 ChatGPT 式逐字输出 |
| 无权限控制 | 单用户场景 | 需要区分"工程师可看"和"管理员可看"时需加 |

## 六、一句话总结

**这是一个面向小规模团队的知识库 Agent——工程师用自然语言提问，Agent 自主检索、推理、回答，不知道就说不知道。核心设计原则：每一步选型都有场景驱动的理由，不是"因为流行所以用"。**
